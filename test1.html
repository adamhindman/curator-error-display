On Visibility: Private vs. Public (vs. Discoverable / Open?)

The project creation picker offers four states: Private, Discoverable, Public, and Open. The question is whether all four should appear as chips in the banner.

Recommendation: show only Private and Public in the chip. Here's the reasoning:

Creation Setting Banner Chip Why
Private Private Clear — project is hidden from search, only invited users can see it.
Discoverable Public From a visitor's perspective, discoverable projects appear in search — they feel public. The access chip (e.g. "Conditions Required") handles the nuance that you can't download freely.
Public Public Same chip. The access chip communicates whether data is freely downloadable or controlled.
Open Public Open = Public + Open Data. The access chip "Open Data" communicates the rest.
Should the owner see a different chip? The owner could potentially see the exact setting they chose (Private / Discoverable / Public / Open) since they care about how their project is configured. Visitors only care about what they can do. If you want this, the owner's chip could show all four states with a settings gear icon that links to sharing settings. But for an easy first pass, Private vs. Public is clean and sufficient.

Data Access Chips — Complete Landscape

Synapse's data access model has these tiers (from Synapse docs). I'm proposing we rename the current search badges slightly to make it clearer these are about data download permissions:

Current Search Badge Proposed Banner Chip Meaning When Shown
Open Access Open Data You can browse and download everything, no conditions. Logged-in user with full download access to an open/public project.
(not currently shown) Conditions Required Data requires meeting access requirements (e.g. signing a DUA, IRB approval). Logged-in user viewing a controlled-access project where conditions for use exist.
Sign In Required Sign In to Download Content is visible but you must create an account / sign in to download. Anonymous user viewing a public project.
Access Denied No Download Access You're logged in but you don't have permission to download. Logged-in user without download ACL on a private/controlled project.
Why rename? The current badges ("Open Access", "Access Denied") are ambiguous — "access" could mean viewing, editing, or downloading. Since the primary gated action is downloading data, making the chips explicitly about download rights reduces confusion. "Open Data" aligns with common scientific data terminology. "No Download Access" is clearer than "Access Denied" which sounds like you can't even view the page. "Conditions Required" surfaces the controlled-access tier that currently has no badge at all.

Scenario Coverage

The six scenarios above attempt to cover the full permission landscape. Here's the matrix:

Scenario User Visibility Access Chip What it tests
A Owner (logged in) Public Open Data Typical PI viewing their own public project
B Owner (logged in) Private hidden Owner viewing their own private project — access chip is redundant (you always have full access to your own project)
C Visitor (logged in) Public Open Data Researcher browsing UKB-PPP — can download freely
D Visitor (logged in) Public Conditions Required Researcher viewing a controlled-access project (e.g. DRI Multi-'omics) — needs to meet requirements
E Visitor (logged in) Private No Download Access Someone who can see the project page (via direct link) but has no download permission
F Anonymous Public Sign In to Download Not signed in, browsing a public project
Are we missing any? One edge case not covered: an anonymous user encountering a private project — but they'd never see the page at all (redirected to sign in), so there's no banner to show. Another edge: a logged-in user who has partial access (can download some files but not others, due to mixed ACLs on sub-folders). For v1, the chip could show the "most restrictive" access level applicable to the project root.
